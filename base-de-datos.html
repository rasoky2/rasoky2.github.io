Explicación de Base de Datos en MariaDB

En este documento se presenta el código de creación de la base de datos en MariaDB para el proyecto de la aplicación web de venta de videojuegos.

#bash #mysql #mariadb #sql #database #JPA

Antes de empezar

  • Se tomó como referencia el modelo de base de datos de la tienda de videojuegos Steam, para crear una base de datos que se adapte a las necesidades de la aplicación web.

  • También el uso de DataGrip como herramienta de base de datos para ejecutar scripts de base de datos en MariaDB.

  • Y el uso de Vaadin como marco de desarrollo de aplicaciones web de Java.

  • Y el uso de MariaDB como gestor de base de datos porque es un sistema de gestión de base de datos relacional que es compatible con MySQL.

  • En vaadin se recomienda usar MariaDB como base de datos, ya que es compatible con MySQL y es fácil de usar, según la información de la documentación oficial de Vaadin.

  • En vaadin se usará JPA para la persistencia de datos, por lo que se creará una base de datos en MariaDB con las tablas necesarias para almacenar la información de los usuarios, los juegos, los DLC, las reseñas, los requisitos, las noticias, los eventos, las compras, los desarrolladores, los amigos, las notificaciones y las compras de los usuarios.

  • El script de creación de la base de datos se ejecutará en MariaDB para crear la base de datos y las tablas necesarias para almacenar la información de los usuarios, los juegos, los DLC, las reseñas, los requisitos, las noticias, los eventos, las compras, los desarrolladores, los amigos, las notificaciones y las compras de los usuarios.

  • Se creará una base de datos llamada vaadinstart con las tablas t_usuario, t_juego, t_dlc, t_reseña, t_subreseña, t_requisitos, t_noticias, t_evento, t_compra, t_desarrollador, t_amigos, t_notificaciones, t_compras.

  • Se creará una jerarquía de tablas para almacenar la información de los usuarios, los juegos, los DLC, las reseñas, los requisitos, las noticias, los eventos, las compras, los desarrolladores, los amigos, las notificaciones y las compras de los usuarios.

Porque MariaDB

  • MariaDB es un sistema de gestión de base de datos relacional que es compatible con MySQL.

  • MariaDB es fácil de usar y es compatible con JPA, que es el estándar de persistencia de Java.

  • MariaDB es compatible con Vaadin, que es un marco de desarrollo de aplicaciones web de Java.

  • En vaadin usar JPA es recomendada por la documentación oficial de Vaadin.

Que es JPA

  • JPA es el estándar de persistencia de Java.

  • JPA es un estándar de Java que define una API para la persistencia de objetos en bases de datos relacionales.

  • JPA es compatible con MariaDB, que es un sistema de gestión de base de datos relacional que es compatible con MySQL.

  • JPA es compatible con Vaadin, que es un marco de desarrollo de aplicaciones web de Java.

  • En vaadin usar JPA es recomendada por la documentación oficial de Vaadin.

Porque vaadinstart

  • vaadinstart es el nombre de la base de datos que se creará en MariaDB.

  • Al parecer, vaadinstart es el nombre de la base de datos que se recomienda usar en Vaadin o ya existe en Vaadin.

  • Se puede usar otro nombre para la base de datos, pero se recomienda usar vaadinstart para seguir las convenciones de Vaadin.

Porque Datagrip

  • Datagrip es una herramienta de base de datos que se puede usar para ejecutar scripts de base de datos en MariaDB.

  • Datagrip tiene una interfaz gráfica que permite ejecutar scripts de base de datos en MariaDB.

  • Datagrip es compatible con MariaDB, que es un sistema de gestión de base de datos relacional que es compatible con MySQL.

  • Datagrip es compatible con JPA, que es el estándar de persistencia de Java.

  • Datagrip es compatible con Vaadin, que es un marco de desarrollo de aplicaciones web de Java.

  • En vaadin usar Datagrip es recomendada por la documentación oficial de Vaadin.

  • El uso de IntelliJ IDEA con el complemento de Datagrip es recomendado por la documentación oficial de Vaadin.

Aquí el código:

CREATE DATABASE IF NOT EXISTS vaadinstart; USE vaadinstart; CREATE TABLE t_usuario ( id_usuario BIGINT PRIMARY KEY AUTO_INCREMENT, nickname VARCHAR(50) NOT NULL, contraseña VARCHAR(255) NOT NULL, nombre VARCHAR(100) NOT NULL, photo_profile VARCHAR(255), descripcion TEXT, redes1 VARCHAR(255), redes2 VARCHAR(255), redes3 VARCHAR(255), redes4 VARCHAR(255), redes5 VARCHAR(255), cumpleaños DATE, saldo DECIMAL(10, 2), wish BIGINT, pais VARCHAR(50), departamento VARCHAR(50), provincia VARCHAR(50), FOREIGN KEY (wish) REFERENCES t_juego(id_juego) ); CREATE TABLE t_juego ( id_juego BIGINT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, photo_game VARCHAR(255), costo DECIMAL(10, 2), descuento DECIMAL(5, 2), tiempo_descuento DATE, trailer VARCHAR(255), trailer2 VARCHAR(255), imagen1 VARCHAR(255), imagen2 VARCHAR(255), imagen3 VARCHAR(255), imagen4 VARCHAR(255), etiqueta TEXT, fecha DATE, idioma VARCHAR(50), dlc BOOLEAN, reddit VARCHAR(255), reseña TEXT ); CREATE TABLE t_dlc ( id_dlcgame BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, costo DECIMAL(10, 2), descuento DECIMAL(5, 2), tiempo_descuento DATE, imagen1 VARCHAR(255), fecha DATE, FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_reseña ( id_reseña BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, id_usuario BIGINT, mensaje TEXT, quality ENUM('Positiva', 'Negativa'), likes INT DEFAULT 0, fecha DATE, FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego), FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario) ); CREATE TABLE t_subreseña ( id_subreseña BIGINT PRIMARY KEY AUTO_INCREMENT, id_reseña BIGINT, mensaje TEXT, fecha DATE, likes INT DEFAULT 0, FOREIGN KEY (id_reseña) REFERENCES t_reseña(id_reseña) ); CREATE TABLE t_requisitos ( id_requisitos BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, tipo ENUM('pesado', 'media', 'ligero'), grapichboard VARCHAR(255), ram VARCHAR(50), procesador VARCHAR(255), SO VARCHAR(255), almacenamiento VARCHAR(255), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_noticias ( id_noticia BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, titulo VARCHAR(255), descripcion TEXT, fecha DATE, tipo ENUM('actualizacion importante', 'periodica', 'noticia', 'evento'), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_evento ( id_evento BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, fecha DATE, descripcion TEXT, video VARCHAR(255), foto VARCHAR(255), foto2 VARCHAR(255), fecha_restante DATE, estado ENUM('terminado', 'en curso'), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_compra ( id_compra BIGINT PRIMARY KEY AUTO_INCREMENT, id_usuario BIGINT, id_juego BIGINT, fecha DATE, FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_desarrollador ( id_desarrollador BIGINT PRIMARY KEY AUTO_INCREMENT, descripcion TEXT, creoeljuego BIGINT, photo VARCHAR(255), redes1 VARCHAR(255), redes2 VARCHAR(255), redes3 VARCHAR(255), redes4 VARCHAR(255), redes5 VARCHAR(255), FOREIGN KEY (creoeljuego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_amigos ( id_usuario BIGINT, id_amigo BIGINT, estado ENUM('pendiente', 'aceptado', 'rechazado', 'bloqueado'), fecha_solicitud DATE, fecha_aceptacion DATE, PRIMARY KEY (id_usuario, id_amigo), FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario), FOREIGN KEY (id_amigo) REFERENCES t_usuario(id_usuario) ); CREATE TABLE t_notificaciones ( id_notificacion BIGINT PRIMARY KEY AUTO_INCREMENT, id_usuario BIGINT, id_juego BIGINT, mensaje TEXT, leido BOOLEAN DEFAULT FALSE, fecha DATE, FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego) ); CREATE TABLE t_compras ( id_compra BIGINT PRIMARY KEY AUTO_INCREMENT, id_juego BIGINT, id_dlcgame BIGINT, id_usuario BIGINT, fecha DATE, costo DECIMAL(10, 2), metodo_pago VARCHAR(50), reembolso BOOLEAN DEFAULT FALSE, FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego), FOREIGN KEY (id_dlcgame) REFERENCES t_dlc(id_dlcgame), FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario) ); CREATE TABLE t_reembolso ( id_reembolso BIGINT PRIMARY KEY AUTO_INCREMENT, id_compra BIGINT, id_dlcgame BIGINT, id_usuario BIGINT, id_juego BIGINT, fecha DATE, motivo ENUM('Error en la compra', 'Error en el juego', 'No cumple con los requisitos', 'No es lo que esperaba', 'Otro'), mensaje TEXT, FOREIGN KEY (id_compra) REFERENCES t_compras(id_compra), FOREIGN KEY (id_usuario) REFERENCES t_usuario(id_usuario), FOREIGN KEY (id_juego) REFERENCES t_juego(id_juego), FOREIGN KEY (id_dlcgame) REFERENCES t_dlc(id_dlcgame) );

Después de ejecutar este script en MariaDB, se creará la base de datos vaadinstart con las tablas t_usuario, t_juego, t_dlc, t_reseña, t_subreseña, t_requisitos, t_noticias, t_evento, t_compra, t_desarrollador, t_amigos, t_notificaciones, t_compras.

Errores

Para evitar errores en la base de datos, se agregarán restricciones a las tablas para que no se puedan insertar valores nulos, negativos o mayores a 100 en los campos de costo, descuento, saldo, nickname, nombre, mensaje, descripción, etc.  ·  Así se evitan errores en la base de datos y se garantiza la integridad de los datos.  ·  Y se evita el uso de condicionales en el código para manejar los errores.

  • Agregar una columna mas a la tabla t_compras para saber si el juego fue reembolsado o no.

ALTER TABLE t_compras ADD COLUMN reembolsado BOOLEAN DEFAULT FALSE;

  • El costo de los juegos y dlcgame no puede ser negativo, por lo que se agrega una restricción para que no se pueda insertar un valor negativo.

ALTER TABLE t_juego ADD CONSTRAINT costo_positivo CHECK (costo >= 0); ALTER TABLE t_dlc ADD CONSTRAINT costo_positivo CHECK (costo >= 0);

  • El descuento de los juegos y dlcgame no puede ser negativo, por lo que se agrega una restricción para que no se pueda insertar un valor negativo.

ALTER TABLE t_juego ADD CONSTRAINT descuento_positivo CHECK (descuento >= 0); ALTER TABLE t_dlc ADD CONSTRAINT descuento_positivo CHECK (descuento >= 0);

  • El descuento de los juegos y dlcgame no puede ser mayor a 100, por lo que se agrega una restricción para que no se pueda insertar un valor mayor a 100.

ALTER TABLE t_juego ADD CONSTRAINT descuento_menor100 CHECK (descuento <= 100); ALTER TABLE t_dlc ADD CONSTRAINT descuento_menor100 CHECK (descuento <= 100);

  • El saldo de los usuarios no puede ser negativo, por lo que se agrega una restricción para que no se pueda insertar un valor negativo.

ALTER TABLE t_usuario ADD CONSTRAINT saldo_positivo CHECK (saldo >= 0);

  • El nickname de los usuarios no puede ser nulo, por lo que se agrega una restriccion para que no se pueda insertar un valor nulo.

ALTER TABLE t_usuario ADD CONSTRAINT nickname_notnull CHECK (nickname IS NOT NULL);

  • El nombre de los juegos no puede ser nulo, por lo que se agrega una restriccion para que no se pueda insertar un valor nulo.

ALTER TABLE t_juego ADD CONSTRAINT nombre_notnull CHECK (nombre IS NOT NULL);

  • El mensaje de las reseñas no puede ser nulo, por lo que se agrega una restriccion para que no se pueda insertar un valor nulo.

ALTER TABLE t_reseña ADD CONSTRAINT mensaje_notnull CHECK (mensaje IS NOT NULL);

ALTER TABLE t_noticias ADD CONSTRAINT descripcion_notnull CHECK (descripcion IS NOT NULL);
